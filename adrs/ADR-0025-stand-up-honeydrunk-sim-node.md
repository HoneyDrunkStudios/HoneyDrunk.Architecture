## Context

`HoneyDrunk.Sim` is cataloged in `catalogs/nodes.json` as the AI sector's simulation-and-plan-evaluation substrate, but the repo is cataloged-not-yet-created — no packages, no contracts, no simulator, no scenario primitive, no plan validator, no target adapter, no CI. Agents runs single agents live per ADR-0020; Flow coordinates multi-step workflows live per ADR-0024; Operator enforces live policy per ADR-0018; Evals scores a target's real behavior after a live run per ADR-0023; none of them own the job Sim is cataloged to own — **predict what would happen** before any live state is touched. Without a dedicated substrate, every Node that needs a dry-run (Flow for "what would this workflow produce," Agents for "would this agent spin up this tool-call sequence," HoneyHub when live for "is this dispatched work safe to commit") either blocks on Sim or invents its own mini-simulator, and the prediction surface fragments across the AI sector.

**This is the 9th and final AI-sector stand-up.** ADR-0016 stood up HoneyDrunk.AI as the inference substrate. ADR-0017 stood up HoneyDrunk.Capabilities as the tool-registry and dispatch substrate. ADR-0018 stood up HoneyDrunk.Operator as the human-policy enforcement and audit substrate. ADR-0020 stood up HoneyDrunk.Agents as the agent-runtime foundation node. ADR-0021 stood up HoneyDrunk.Knowledge as the external-information ingestion and retrieval substrate. ADR-0022 stood up HoneyDrunk.Memory as the agent-memory substrate. ADR-0023 stood up HoneyDrunk.Evals as the evaluation and quality substrate. ADR-0024 stood up HoneyDrunk.Flow as the workflow-orchestration substrate. Sim is the last of the nine cataloged AI-sector Nodes. Every other AI-sector Node has been shaped by a stand-up ADR; Sim closes the wave.

The stand-up pattern is deliberately reused: contracts live in an `Abstractions` package, runtime composition is a separate package, downstream Nodes compile against `Abstractions` only, and a first-wave `Providers.InMemory` package ships at stand-up so consumers have one shared in-memory backend for scenario execution that they can compose in tests and in local development without standing up a production simulation backend. The `.Providers.InMemory` name is chosen over `.Testing` deliberately — it matches the shape Knowledge (ADR-0021 D2), Memory (ADR-0022 D2), Evals (ADR-0023 D2), and Flow (ADR-0024 D2) took, and it signals that the in-memory backend is a production-shaped provider-slot implementation on Sim's scenario-execution-backend axis, not a test-only artifact. The `.Testing` pattern ADR-0017 D2 applied to Capabilities remains valid for that Node (where there is no provider-slot axis at the registry layer), but it is not the right shape for Sim, which has a clear provider-slot family on the scenario-execution side (rule-based, model-backed via `IChatClient`, shared in-memory, replayed-transcript — any of which could ship later under the `Providers.*` family).

The `catalogs/contracts.json` entry for `honeydrunk-sim` currently lists two interfaces (`ISimulator`, `ISimulationResult`), while the `catalogs/relationships.json` `exposes.contracts` list for the same Node names four interfaces (`ISimulator`, `IScenario`, `IRiskAssessment`, `IPlanValidator`), and the repo overview at `repos/HoneyDrunk.Sim/overview.md` names the same four. Three different sources, three different contract sets, all seeded before any ADR pinned the Sim shape. This is the same three-way drift Evals faced at ADR-0023 Context, and it is reconciled here by pinning the D3 definitive set (six surfaces — three interfaces plus three records) against every cataloged and prose reference. Interfaces retain the `I` prefix and are cataloged with `kind: "interface"`; records drop the `I` prefix and are cataloged with `kind: "type"`, matching the grid-wide naming rule already applied by `ModelCapabilityDeclaration` (ADR-0016), `CapabilityDescriptor` (ADR-0017), the four governance records (ADR-0018), `KnowledgeSource` (ADR-0021), and the record promotions at ADR-0023 D3 (`EvalCase`, `EvalReport`).

Sim's boundary against the adjacent Nodes needs explicit disambiguation before drift creeps in, and three particular concerns — the Sim-vs-Evals boundary, the router-bypass pattern for deterministic model selection, and the Sim-vs-Flow execute-vs-dry-run boundary — need carving at stand-up rather than left to a "figure it out later" escape hatch.

1. **Sim vs Evals.** Evals consumes a target's real behavior to score it; Sim substitutes for a target's real behavior to predict what would happen. The primitive that separates the two is **side-effect-freedom**. `repos/HoneyDrunk.Sim/invariants.md` item 2 already states "simulations have no side effects — a simulation never modifies real state, no database writes, no message sends, no tool executions against live systems," and this ADR elevates that to the cross-Node boundary test: **if it writes to any Grid-durable surface, it is not Sim.** Evals reads what the target actually did; Sim predicts what the target would do. D6 pins the rule.
2. **Sim vs AI.** AI owns inference (`IChatClient`, `IEmbeddingGenerator`, `IModelProvider`, `ModelCapabilityDeclaration` per ADR-0016). Sim does not re-implement inference. But Sim has a reproducibility hazard AI's router-centric surface does not directly serve: a simulation that selects different models on different runs cannot be reproduced for investigation or for cross-run comparison. A risk assessment that depended on "whatever the router picked today" is not an assessment; it is a coin flip. The question then is how Sim pins a specific `IModelProvider` / `ModelCapabilityDeclaration` for a simulation run without widening `IChatClient` and without duplicating the routing layer. This ADR mirrors the ADR-0023 D6 pattern by giving Sim its own target abstraction, `ISimulationTarget`, parallel to Evals's `IEvalTarget`. D8 pins the router-bypass shape.
3. **Sim vs Flow.** Flow executes multi-step workflows live per ADR-0024 D1 — it runs them, persists state, compensates partial failures, and resumes after a pause. Sim dry-runs the same workflow shape against deterministic fixtures without any of those side effects. The boundary is the execute-vs-dry-run distinction that `repos/HoneyDrunk.Sim/boundaries.md` decision test already records; this ADR pins the Sim side at the runtime level and names the specific contract consumed (`IWorkflow`, read-only). ADR-0024 D4/D6 pinned the Flow side at the boundary level ("Sim consumes `IWorkflow` definitions as read-only fixtures"); this ADR pins the reciprocal edge. D4 records the rule and calls out that Sim never invokes `IWorkflowEngine` — it walks `IWorkflow` definitions through its own simulation substrate.
4. **Sim vs Operator.** Operator enforces live policy per ADR-0018. Sim does not enforce. But a plan validator that ignored cost, safety, and approval concerns would produce risk assessments that are structurally useless — a "plan looks fine" verdict that would be blocked by `ICostGuard` or `IApprovalGate` in a real run is the plan validator's failure, not a trivia. Sim composes `ICostGuard`, `ISafetyFilter`, and `IApprovalGate` from `HoneyDrunk.Operator.Abstractions` as **observation-only primitives** to predict what Operator would do in a real execution. This mirrors the ADR-0023 D7 Evals-style Operator composition: the distinction is observes-vs-enforces, pinned here at the Sim boundary. D9 records the rule.
5. **Sim vs Memory.** Sim needs seeded fixtures for scenario execution — a deterministic starting memory state, a deterministic starting knowledge set, a deterministic starting agent context — without touching the real Memory or Knowledge stores. Memory's `Providers.InMemory` per ADR-0022 D2 is the fixture-composition primitive; Sim composes `Providers.InMemory` to seed scenario state without engaging a production Memory backend. D11 pins the fixture-composition rule.

This ADR is the **stand-up decision** for the Sim Node — what it owns, what it does not own, which contracts it exposes, how downstream Nodes couple to it, and how it interacts with AI, Capabilities, Operator, Agents, Knowledge, Memory, Evals, Flow, Lore, Pulse, and HoneyHub. "Node" is used throughout in the ADR-0001 sense — a library-level building block producing one or more NuGet packages, not a deployable service. This ADR is not a scaffolding packet. Filing the repo, adding CI, wiring the in-memory provider, and producing the first shippable packages all follow as separate issue packets once this ADR is accepted.

## Decision

### D1. HoneyDrunk.Sim is the AI sector's simulation and plan-evaluation substrate

`HoneyDrunk.Sim` is the single Node in the AI sector that owns **simulation primitives** — the contracts and runtime machinery that define a scenario, run it against a target substitute, produce a projected outcome, score the projection for risk, and validate a proposed plan before any live side effect happens. It is a shared substrate, not an application. It does not decide *what scenarios to simulate* or *what risk threshold is acceptable*; it owns the mechanics of *how a scenario is defined, how a target is substituted for a real behavior under simulation, how a projection is produced, and how the projected outcome is surfaced as a risk-assessed simulation result*.

Simulation *content* — the actual scenarios, fixtures, risk rubrics, and acceptance thresholds — is decided by the consumers. Each consumer maintains its own scenarios for the concerns it cares about: Flow can ship workflow-dry-run scenarios that walk `IWorkflow` definitions without invoking `IWorkflowEngine`; Agents can ship agent-plan scenarios that predict tool-call sequences; HoneyHub when live can ship dispatched-work scenarios that validate a commit-to-real-execution decision; application Nodes can ship their own domain-specific risk scenarios. Sim provides the substrate; consumers provide the content.

Sim is the closing Node of the AI-sector stand-up wave. The three substrates (AI, Capabilities, Operator) plus the foundation triad (Agents + Knowledge + Memory) plus Evals and Flow gave the Grid the primitives to run, evaluate, and orchestrate AI workloads. Sim is the primitive that lets the Grid **ask "what would happen"** without committing to the side effects of finding out live. The foundation is not "complete" without this primitive — every downstream Node that wants a preview before commit would otherwise either block or invent a local simulator, and the divergence across those local simulators would be immediate.

### D2. Package families

The Sim Node ships the following package families, mirroring the stand-up shape used by ADR-0016 (AI), ADR-0017 (Capabilities), ADR-0018 (Operator), ADR-0019 (Communications), ADR-0020 (Agents), ADR-0021 (Knowledge), ADR-0022 (Memory), ADR-0023 (Evals), and ADR-0024 (Flow):

- `HoneyDrunk.Sim.Abstractions` — all interfaces (`ISimulator`, `IPlanValidator`, `ISimulationTarget`) and all records (`Scenario`, `RiskAssessment`, `SimulationResult`) per D3. Zero runtime dependencies beyond `HoneyDrunk.Kernel.Abstractions`, `HoneyDrunk.AI.Abstractions` (for `IChatClient`, `IModelProvider`, and `ModelCapabilityDeclaration` on the simulation-target path per D8), and `HoneyDrunk.Flow.Abstractions` (for read-only `IWorkflow` consumption per D4 and D5). Crucially, `HoneyDrunk.Sim.Abstractions` does **not** take a compile-time dependency on `HoneyDrunk.Operator.Abstractions` or `HoneyDrunk.Memory.Abstractions` — both of those are runtime-only edges consumed inside the default `HoneyDrunk.Sim` package (see D9 and D11). Consumers of `Abstractions` do not inherit Operator or Memory transitively.
- `HoneyDrunk.Sim` — runtime composition: default `ISimulator` with scenario execution, default `IPlanValidator` with rule-based risk scoring, default `ISimulationTarget` implementations including a chat-backed target that wraps `IChatClient` and optionally pins a `ModelCapabilityDeclaration`, reproducibility primitives per D7, DI wiring. Takes first-class runtime dependencies on `HoneyDrunk.AI.Abstractions` (for chat-backed targets and model pinning), `HoneyDrunk.Flow.Abstractions` (for workflow-definition walking), `HoneyDrunk.Operator.Abstractions` (for `ICostGuard`, `ISafetyFilter`, `IApprovalGate` observation-only composition per D9), and `HoneyDrunk.Memory.Abstractions` plus `HoneyDrunk.Memory.Providers.InMemory` (for fixture composition per D11).
- `HoneyDrunk.Sim.Providers.InMemory` — in-memory scenario-execution backend. Deterministic, side-effect-free by construction, suitable for tests, local development, and seed simulation runs. Consumed by downstream Nodes in test projects and in local composition; never in production composition of a real risk assessment.

A production-grade scenario-execution backend — a dedicated model-backed simulator, a replay-transcript provider, a rule-augmented hybrid — is **not first-wave**. The provider slot exists at stand-up via `Providers.InMemory`; a production backend ships under a separate issue packet once a real consumer (Flow's workflow-dry-run scenarios, HoneyHub's dispatched-work validation when live, or the first production plan-validator pipeline) drives the shape. This is the same staged-shape pattern ADR-0021 D2, ADR-0022 D2, ADR-0023 D2, and ADR-0024 D2 applied to provider slots. Pinning the side-effect-freedom principle at the contract level (D6) is independent of committing to a specific scenario-execution substrate.

No separate `HoneyDrunk.Sim.Testing` package is introduced at stand-up. The `Providers.InMemory` package already plays that role — it is the in-memory backend every downstream test project can compose. This differs from Capabilities (ADR-0017 D2), which ships `.Testing` because there is no provider-slot axis at the registry layer; Sim, like Knowledge, Memory, Evals, and Flow, has a clear provider-slot axis (scenario-execution backend) and the in-memory fixture belongs there.

### D3. Exposed contracts

Six surfaces form the Sim Node's public boundary at stand-up — three interfaces and three records. These are the surfaces downstream Nodes are allowed to compile against, and they are the definitive set against which the three drifted cataloged sources (`contracts.json`, `relationships.json`, `repos/HoneyDrunk.Sim/overview.md`) are reconciled per the follow-up work section.

| Contract | Kind | Purpose |
|---|---|---|
| `ISimulator` | interface | Run a `Scenario` against an `ISimulationTarget`, collect the projected outcome, produce a `SimulationResult`. The top-level orchestration surface. |
| `IPlanValidator` | interface | Validate a proposed plan (a `Scenario` paired with a candidate execution shape) before real execution. Composes `ISimulator` internally; returns a `RiskAssessment` plus a go/no-go verdict. The plan-evaluation entry point. |
| `ISimulationTarget` | interface | The thing being simulated — a chat model, a workflow definition, an agent, a retrieval pipeline. Abstracts over what the scenario is run against without binding Sim to any consumer's runtime surface. Parallel in shape to Evals's `IEvalTarget` per ADR-0023 D3; does **not** share the `IEvalTarget` type — Sim owns `ISimulationTarget` to avoid creating a Sim → Evals runtime edge. See D8. |
| `Scenario` | record | A single simulation scenario — initial state fixture, proposed actions or workflow under test, constraints, per-scenario metadata (identity, version, tags). Value type; `kind: "type"` in the catalog. Replaces the previous prose-level `IScenario`. |
| `RiskAssessment` | record | Structured risk evaluation — identified failure modes, per-mode probability, per-mode mitigation, confidence level per `repos/HoneyDrunk.Sim/invariants.md` item 3, aggregate severity. Value type; `kind: "type"` in the catalog. Replaces the previous prose-level `IRiskAssessment`. |
| `SimulationResult` | record | Structured simulation results — projected outcome payload, per-step trace, observed Operator-primitive firings (from D9), a `RiskAssessment`, run provenance (target identity, `ModelCapabilityDeclaration` identity when pinned, scenario version, timestamps, reproducibility seed per D7). Value type; `kind: "type"` in the catalog. Replaces the previous interface-level `ISimulationResult`. |

`Scenario`, `RiskAssessment`, and `SimulationResult` are new record entries at the catalog level and apply the grid-wide naming rule (no `I` prefix, `kind: "type"`). `ISimulationTarget` is a new interface entry introduced by this ADR per D8. `IPlanValidator` was in `relationships.json` and the repo overview but absent from `contracts.json` — it is added to `contracts.json` per the follow-up work section. `ISimulationResult` (from `contracts.json`) and `IScenario`/`IRiskAssessment` (from `relationships.json` and the repo overview) are promoted to records and renamed without the `I` prefix. No `kind: "record"` is used anywhere in the catalog schema.

Concrete `ISimulationTarget` implementation shapes — a `WorkflowTarget` wrapping an `IWorkflow` definition, an `AgentTarget` wrapping an `IAgent` plan, a `RetrievalTarget` wrapping an `IRetrievalPipeline`, a transcript-replay target — are **deferred to the scaffold packet**. The stand-up commitment is the `ISimulationTarget` interface itself plus a default chat-backed target shape (wrapping `IChatClient` with optional `ModelCapabilityDeclaration` pinning per D8). The generalisation over workflow / agent / retrieval targets is pinned at the contract level in D4 and D8; the concrete target shapes land with scaffold. This matches the deferral pattern ADR-0020 D3 used for agent record shapes, ADR-0021 D3 used for retrieval request/response shapes, ADR-0022 D3 used for `MemoryEntry`, ADR-0023 D3 used for concrete `IEvalTarget` shapes, and ADR-0024 D3 used for workflow record shapes.

Monte Carlo / N-trial distribution surfaces (run the same scenario N times and surface a distribution of outcomes) are also **deferred to scaffold**. The stand-up commitment is the single-trial `ISimulator` shape in D3; distribution-surface APIs are a scaffold concern driven by the first real consumer that needs them.

### D4. Boundary rule with AI, Capabilities, Operator, Agents, Knowledge, Memory, Evals, Flow, Lore, Pulse, and HoneyHub (definitive)

The surrounding Nodes need an explicit boundary test so that drift does not creep in once Sim ships. The rule below is the decision test Sim applies when an ambiguous concern is proposed.

**Decision test — for any concern in the simulation path, ask:**

1. Does it make a live inference call against a model to produce a live output? → **AI** (Sim composes `IChatClient`, `IModelProvider`, and `ModelCapabilityDeclaration` per D5 and D8 for target substitution; it never talks to providers directly).
2. Does it discover or invoke a live tool with side effects? → **Capabilities** (when a simulated target would invoke a tool in real execution, Sim records the intended invocation in the `SimulationResult` trace but does **not** dispatch through `ICapabilityInvoker` — that would produce a side effect, violating D6).
3. Does it enforce a live production policy — block a response that violates safety, block a request that exceeds a cost ceiling, write an audit record of a human approval? → **Operator** (Sim composes Operator primitives per D9 as observation-only predictions of what Operator would do; Sim never enforces).
4. Does it manage agent identity, lifecycle, or execution context in a live execution? → **Agents** (when the target is an agent shape, Sim substitutes for the agent's runtime behavior through `ISimulationTarget`; it does not invoke `IAgent` live).
5. Does it ingest or retrieve externally-sourced information with live side effects? → **Knowledge** (when a simulated target would retrieve documents in real execution, Sim composes seeded knowledge fixtures per D11 rather than engaging a production retrieval pipeline).
6. Does it persist agent-generated content across executions? → **Memory** (Sim composes `Providers.InMemory` fixtures for scenario seeding per D11; Sim never writes to a production Memory store).
7. Does it score a target's **real** output against a rubric or detect regression across **real** runs? → **Evals** (Evals consumes a target's real behavior per ADR-0023 D4/D6; Sim substitutes for that behavior to predict it. The primitive is side-effect-freedom per D6).
8. Does it execute a multi-step workflow live, persist workflow state, compensate partial failures, or resume after a pause? → **Flow** (Sim consumes `IWorkflow` definitions as **read-only fixtures** per D5 and walks them through its own simulation substrate; Sim never invokes `IWorkflowEngine`).
9. Does it plan work at the org-timescale — decompose goals into features, features into tasks? → **HoneyHub** (HoneyHub decides *what* to simulate and *whether the simulation's verdict is acceptable*; Sim runs the simulation).
10. Does it ingest telemetry or host signal streams? → **Pulse** (Sim emits simulation lifecycle telemetry into Pulse per D10 but has no runtime dependency on Pulse, matching the emit-only stance every prior AI-sector stand-up took).
11. Does it define a scenario, substitute for a target's real behavior to predict what would happen, score the projection for risk, and validate a plan before any live side effect? → **Sim**.

Under this test, several subtleties are worth naming explicitly:

- **The primitive that separates Sim from Evals is side-effect-freedom.** Evals consumes a target's real behavior to score it — every real-execution side effect the target would produce in production (Operator writes, Memory writes, Communications sends, Flow state checkpoints) happens during evaluation. Sim substitutes for that real behavior to predict what would happen — none of those side effects fire. The rule is pinned in D6: if it writes to any Grid-durable surface, it is not Sim.
- **Sim is a read-only consumer of Flow definitions, not of Flow runtime.** The reciprocal edge ADR-0024 D4/D6 pinned at the Flow side is completed here at the Sim side. Sim walks `IWorkflow` and `IWorkflowStep` as read-only authoring surfaces; it never invokes `IWorkflowEngine.StartAsync()`. A simulation of a workflow is a walk over the definition with substituted target behavior, not an execution of the engine.
- **Sim composes Operator primitives as observation-only predictions.** The ADR-0023 D7 pattern (Evals observes Operator without enforcing) applies here with a slightly different framing: Sim observes Operator to *predict what Operator would do*. When a simulated target would emit an output, Sim consults `ISafetyFilter` to predict whether Operator would block it, `ICostGuard` to predict whether Operator would halt on cost, `IApprovalGate` to predict whether Operator would pause for approval. Sim never invokes these to enforce; it invokes them to predict. The distinction is load-bearing for D9.
- **The router-bypass precedent applies.** ADR-0023 D6 established that a Node that needs to pin a specific `ModelCapabilityDeclaration` does so through a Node-owned target abstraction, not by widening `IChatClient` and not by duplicating `IModelProvider` composition. Sim follows the same pattern: `ISimulationTarget` is Sim's router-bypass boundary per D8, and it parallels `IEvalTarget` in shape without sharing the type. Sharing the type would create a Sim → Evals runtime edge that neither Node needs — Sim does not evaluate and Evals does not simulate. Each Node owns its own target abstraction.

`catalogs/relationships.json` currently lists Sim's `consumes` as `honeydrunk-ai`, `honeydrunk-knowledge`, `honeydrunk-agents`. Under this ADR, the list widens to include `honeydrunk-kernel` (currently missing), `honeydrunk-flow` (reciprocal to ADR-0024 D4/D6), `honeydrunk-memory` (for fixture composition per D11), and `honeydrunk-operator` (for observation-only primitive composition per D9), and the `honeydrunk-ai` `consumes_detail` widens to include `IModelProvider` and `ModelCapabilityDeclaration` alongside `IChatClient` and the package reference. The corrections are tracked in the follow-up work section at the top of this ADR.

### D5. Flow consumption — Sim reads `IWorkflow`, does not invoke `IWorkflowEngine`

`ISimulationTarget`'s workflow-target adapter (concrete shape deferred to scaffold) takes a first-class compile-time dependency on `HoneyDrunk.Flow.Abstractions` and consumes `IWorkflow` and `IWorkflowStep` per ADR-0024 D3 **as read-only authoring surfaces**. Sim walks the workflow definition — step identity, step ordering, transitions, compensation mapping — and simulates each step's outcome against the appropriate `ISimulationTarget`. It does not invoke `IWorkflowEngine.StartAsync()`, does not persist `IWorkflowState`, and does not trigger `ICompensation` executions.

The reciprocal edge to ADR-0024 D4/D6 is closed here: Flow's side pinned "Sim consumes `IWorkflow` definitions as read-only fixtures," and this ADR pins the Sim side at the runtime level and names the specific surfaces consumed (`IWorkflow`, `IWorkflowStep`, `HoneyDrunk.Flow.Abstractions`). The compile-time dependency from `HoneyDrunk.Sim.Abstractions` on `HoneyDrunk.Flow.Abstractions` is accepted — downstream consumers of `HoneyDrunk.Sim.Abstractions` inherit the transitive reference to `HoneyDrunk.Flow.Abstractions`, matching the way ADR-0024 D5 treated the upstream `HoneyDrunk.Agents.Abstractions` edge. The Flow runtime package is **not** transitively dragged in — only the abstractions.

When a simulation needs to model what the Flow engine itself would do at the control-plane level (pause-for-approval behavior, compensation walk on step failure, resume-after-event timing), Sim models that behavior through its own simulation logic, not by invoking the engine. This is consistent with D4's "Sim substitutes for a target's real behavior" framing: the thing being simulated is the engine's behavior, not the engine's live execution path.

### D6. Side-effect-freedom is the Sim–Evals boundary primitive

The cross-Node rule that pins the Sim–Evals boundary is: **if it writes to any Grid-durable surface, it is not Sim.** Evals reads what the target actually did (its real execution produced real Operator writes, real Memory writes, real Communications sends, real Flow state checkpoints); Sim predicts what the target would do (none of those side effects fire during the simulation). This elevates `repos/HoneyDrunk.Sim/invariants.md` item 2 ("simulations have no side effects") from a Node-local invariant to a Grid-level one.

The rule is enforceable at the interface level and at the runtime level:

- **Interface level.** `ISimulator.SimulateAsync(Scenario, ISimulationTarget)` returns a `SimulationResult`. It does not take writer surfaces (no `IMemoryStore`, no `ICommunicationOrchestrator`, no `IWorkflowEngine`, no `IAuditLog`) as parameters and does not compose writer surfaces in its default runtime path.
- **Runtime level.** The default `ISimulationTarget` composition uses `HoneyDrunk.Memory.Providers.InMemory` for memory-state seeding (per D11) rather than production Memory, uses in-memory scenario execution rather than live retrieval, and composes `ICostGuard`/`ISafetyFilter`/`IApprovalGate` observation-only per D9. The provider family is a seed-and-simulate family by construction.

The rule excludes **GridContext flagging** from the side-effect-freedom count. `repos/HoneyDrunk.Sim/invariants.md` item 5 ("GridContext distinguishes simulation from real execution") already requires a simulation-run flag on context so downstream systems can differentiate sim traffic; setting that flag is not a side effect in the durable-surface sense — it is a context marker that rides with the scoped operation and expires with it. Pulse telemetry emitted by the simulation itself (D10) is not a side effect in the D6 sense either — Pulse observes all emitters and does not itself gate the simulation's correctness.

Side-effect-freedom is the primitive that answers every ambiguous Sim question. "Can Sim write an audit entry?" No — Operator owns the audit trail and a write would violate D6. "Can Sim persist a `SimulationResult`?" The artifact is durable from the consumer's perspective if they store it, but Sim itself does not own the durable store at stand-up — consumers choose where to put it, and the default in-memory backend is provider-slot compatible but not a production substrate. "Can Sim invoke an `IWorkflowStep.ExecuteAsync()` live against a real consumer?" No — Sim walks the definition and substitutes the target's behavior; it does not execute.

### D7. Reproducibility — `ISimulationTarget` pins the run, `SimulationResult` records the seed

Every `SimulationResult` record carries a reproducibility seed — a deterministic identifier (seed value, version tag, pinned `ModelCapabilityDeclaration` identity, fixture hash) that lets a consumer re-run the same scenario against the same target and produce the same projected outcome. This is not advisory metadata; it is a contract field on the record, matching the role `ModelCapabilityDeclaration` identity plays for Evals per ADR-0023 D12.

Reproducibility is the foundation of Sim's utility. A risk assessment that cannot be reproduced is an opinion, not a projection — a consumer who sees "this plan has a 0.8 failure-mode probability" needs to be able to re-run the exact scenario and confirm the number before they block or commit a live execution. The reproducibility seed plus the pinned `ModelCapabilityDeclaration` (for chat-backed targets per D8) gives consumers the foundation for that confirmation.

The **reproducibility primitives** — how the seed is computed, how it threads through `ISimulationTarget`, how non-deterministic targets (for example, a live model-backed target in chat mode) are flagged versus deterministic targets (for example, a transcript-replay target) — are a **scaffold-packet concern in their concrete shape**, but the principle is pinned at stand-up: every `SimulationResult` records the seed, and the `ISimulationTarget` interface exposes a deterministic-vs-non-deterministic flag. A non-deterministic target (model-backed without a fixed seed, live retrieval under simulation) is legal under the contract, but it is labelled non-deterministic and that label propagates into the `SimulationResult`. This matches the deterministic-vs-non-deterministic flag Evals applies to `IEvalScorer` per ADR-0023 D3.

### D8. `ISimulationTarget` is the router-bypass boundary — Sim pins models; AI's router is not widened

Reproducibility requires pinning a specific `IModelProvider` / `ModelCapabilityDeclaration` for a simulation run. Running the same scenario against "whatever the router picks today" would defeat the point of simulating — a projection whose underlying model changed silently between runs is not a projection, it is noise. The question then is *where* the pinning primitive lives: widen `IChatClient` to take an explicit-model overload (Option A, rejected), have Sim compose `IModelProvider` directly (Option B, rejected), reuse Evals's `IEvalTarget` (rejected — would create a Sim → Evals runtime edge that neither Node needs), or introduce an `ISimulationTarget` abstraction that owns the pinning (Option C, chosen).

**Option A (widen `IChatClient`) is rejected.** Same reasoning as ADR-0023 D6 Option A. `IChatClient` is the Grid's canonical chat-completion surface and is frozen under ADR-0016 D8's contract-shape canary. Adding an explicit-model overload would (a) break the canary or require an expensive version bump on the hottest interface in the Grid, (b) inflate every caller's API surface with a concern only Sim has, and (c) create a second model-selection path that competes with the routing layer's single source of truth. One Node's special case should not reshape the Grid's hottest interface.

**Option B (Sim composes `IModelProvider` directly) is rejected.** Same reasoning as ADR-0023 D6 Option B. Pulling `IModelProvider` composition into Sim's runtime would either duplicate the router's provider-discovery logic or introduce a second composition path that sees models the router does not see. Both produce drift, and both give Sim a routing-layer responsibility that belongs in AI.

**Option C (`ISimulationTarget` abstraction) is chosen.** `ISimulationTarget` is the Sim-side abstraction over "the thing being simulated." For a chat-backed target, the target pins a `ModelCapabilityDeclaration` and, at invocation time, composes `IModelProvider` (from `HoneyDrunk.AI.Abstractions`, sourced through the host's DI container the same way AI's runtime package resolves providers) for that specific capability declaration. This is a legitimate use of `IModelProvider` — it is what the router itself composes — but it does *not* widen `IChatClient` and does *not* duplicate routing policy. The target is a simulation-time harness, explicitly labelled as bypassing the router, and the `ModelCapabilityDeclaration` identity it pins is recorded on every `SimulationResult` (D3) so the bypass is auditable.

**Reusing Evals's `IEvalTarget` is explicitly rejected.** A shared target type would require `HoneyDrunk.Sim.Abstractions` to take a compile-time dependency on `HoneyDrunk.Evals.Abstractions`, creating a Sim → Evals runtime edge on top of the one each Node's runtime package would inherit from Evals's other dependencies. Sim does not evaluate and Evals does not simulate — the two Nodes do not compose — and introducing the edge would produce a coupling neither Node benefits from. The shapes are parallel but the types are distinct. `ISimulationTarget` in Sim, `IEvalTarget` in Evals; each Node owns its abstraction.

The router-bypass is narrow and deliberate, and it is the only such bypass in the Grid alongside Evals's `IEvalTarget`. Every production inference call outside the two target paths goes through `IChatClient` and whatever router composition is wired.

### D9. Operator composition as observation-only prediction — Sim observes, does not enforce

`ISimulator` and `IPlanValidator` implementations may compose three Operator primitives from `HoneyDrunk.Operator.Abstractions` per ADR-0018 as **observation-only prediction inputs**:

- `ISafetyFilter` — if the simulated target produced this output in a real run, would Operator block it? What rule would fire? Sim records the predicted firing on the `SimulationResult` trace and factors it into the `RiskAssessment`.
- `ICostGuard` — if the simulated target ran at this cost in a real run, would Operator halt on cost? What threshold would be crossed? Sim records the prediction and factors it into the assessment.
- `IApprovalGate` — if the plan would require human approval in a real run, where in the workflow would the pause happen and what would be escalated? Sim records the predicted pause point and surfaces it on the `SimulationResult`.

This mirrors ADR-0023 D7's Evals-style Operator composition with a shift in framing: Evals *observes* what Operator actually did during a real run; Sim *predicts* what Operator would do in a real run. The Operator primitives are the same; the framing is different.

Sim does not *invoke* `ISafetyFilter` to *block* an output — it invokes it to *observe what the filter would return* and record the observation. Sim does not *write* to `IAuditLog` — the audit trail belongs to live execution, not to prediction. Sim does not *raise* an approval request through `IApprovalGate` — it predicts where a real execution would raise one.

The distinction — observes vs enforces — is load-bearing for D4's "Sim is an observer, not an enforcer" rule (analogous to ADR-0023 D4's equivalent rule for Evals). Wiring the Operator primitives as observable prediction inputs gives consumers the plan-validation story for "would this plan trigger a safety block / cost halt / approval pause in real execution" without entangling Sim with Operator's production-enforcement path.

### D10. Simulation telemetry — Pulse consumes, Sim does not depend; metadata-only content

Sim emits telemetry for every simulation lifecycle event (scenario start, scenario complete, scenario fail, predicted safety-filter firing, predicted cost-halt, predicted approval pause), every per-step trace entry (step identity, target identity, predicted outcome, duration, per-step non-determinism flag), and every `RiskAssessment` aggregate (failure-mode count, aggregate severity, confidence level) via Kernel's `ITelemetryActivityFactory`. Pulse consumes that telemetry downstream. **Sim has no runtime dependency on Pulse.** The direction is one-way by contract: Sim emits, Pulse observes. Same rule as every prior AI-sector stand-up.

The content-in-telemetry posture follows the content-telemetry pattern ADR-0021 D10 (Knowledge) and ADR-0022 D9 (Memory) pinned: simulation telemetry is **metadata-only** (identities, outcomes, durations, confidences, predicted-firing markers) and does not carry scenario input content, target output content, or predicted-output payload content. This **differs from the deliberate eval-signal carve-out ADR-0023 D10 established** (eval signals may carry prompts and outputs unless the suite declares itself sensitive, for regression-diagnosis reasons). Sim's telemetry does not face the same regression-diagnosis pressure — simulation projections are forward predictions, not diagnostic artifacts that need payload content for triage. A Pulse signal that scenario X produced a risk score of 0.72 is actionable without the scenario's input text; the full projection is recoverable through the `SimulationResult` if an investigator needs the payloads.

The Memory/Knowledge rule is the conservative default and it is the right one for Sim's payload profile. If a specific use case (for example, a dedicated simulation-signal-review workflow that needs payload content at the telemetry layer) surfaces later, the right path is a scaffold-time opt-in flag analogous to Evals's sensitivity flag, not a stand-up-time carve-out. This is flagged in Alternatives Considered against the "match Evals carve-out" option.

Pulse signal ingress back into Sim — reactive scenario re-selection, reactive risk-threshold tuning, reactive re-run based on observed production telemetry — is out of scope for stand-up. It is flagged in Alternatives Considered as a deferred concern and matches the emit-only stance every prior AI-sector stand-up took.

### D11. Fixture composition — Sim seeds Memory and Knowledge via `Providers.InMemory`, never production

Sim's simulation targets need deterministic starting state — a memory state that reflects what an agent would remember at scenario start, a knowledge set that reflects what retrieval would return at scenario start, an agent context that reflects where in a multi-step plan the simulation begins. Reaching into the production Memory or Knowledge backends to read that state (or, worse, to seed it) would violate D6's side-effect-freedom rule and would entangle simulation with production data.

Sim resolves this by composing **`HoneyDrunk.Memory.Providers.InMemory` per ADR-0022 D2** and **`HoneyDrunk.Knowledge.Providers.InMemory` per ADR-0021 D2** for fixture composition. A `Scenario` carries (or references) the fixture state it needs; Sim's runtime composes the in-memory provider with that fixture at simulation-start and discards it at simulation-end. The production Memory and Knowledge backends are **never** on Sim's runtime edge.

Under this rule:

- Sim's runtime takes a compile-time dependency on `HoneyDrunk.Memory.Abstractions` (for `IMemoryStore`, `IMemoryScope` interfaces the fixture exposes), but consumers of `HoneyDrunk.Sim.Abstractions` do not inherit that dependency transitively — the Memory edge lives in the runtime package only, same split ADR-0024 D8 applied to Flow's Communications edge.
- The `honeydrunk-sim → honeydrunk-memory` edge in `catalogs/relationships.json` is a runtime edge, and its `consumes_detail` explicitly names `Providers.InMemory` alongside the abstractions to signal that the composition is fixture-composition, not production-Memory composition.
- The `honeydrunk-sim → honeydrunk-knowledge` edge (already present) is reinterpreted under this rule: the existing `consumes` entry stands, but the edge is documented as fixture-composition rather than production-retrieval on its `consumes_detail`.

This is consistent with ADR-0023 D9's analogous rule for Evals (Evals seeds Memory and Knowledge `Providers.InMemory` for eval fixtures without engaging the production backends). Sim is a second consumer of the same primitive.

### D12. `SimulationResult` carries full run provenance — target identity, `ModelCapabilityDeclaration`, scenario version, seed, timestamps

Every `SimulationResult` record carries the complete provenance of the run it describes: the scenario identity and version, the `ISimulationTarget` identity, the `ModelCapabilityDeclaration` identity (when the target pinned one per D8), the reproducibility seed (per D7), the run start and end timestamps, the observed Operator-primitive predictions (per D9), the per-step trace, and the `RiskAssessment` payload. This is not advisory metadata; it is a contract field on the record, matching the role `KnowledgeSource` plays for Knowledge per ADR-0021 D6, the embedding-model identifier plays for Memory per ADR-0022 D10, and the `ModelCapabilityDeclaration` identity plays for Evals per ADR-0023 D12.

Provenance is what makes a `SimulationResult` consumable as a decision input. A plan validator that returns a risk verdict without recording what model, what scenario version, what fixture, and what seed produced the verdict is giving a consumer an opinion — not a projection they can act on. The provenance field is the primitive that makes the projection auditable and the verdict defensible.

### D13. Contract-shape canary

A contract-shape canary is added to the Sim Node's CI: it fails the build if any of the following four surfaces change shape (method signatures, parameter shapes, record members) without a corresponding version bump:

- `ISimulator`
- `IPlanValidator`
- `ISimulationTarget`
- `SimulationResult`

These four are the hot path for every real consumer. `ISimulator` is every simulation-running caller's entry point; `IPlanValidator` is the plan-evaluation entry point every commit-to-real-execution gate depends on; `ISimulationTarget` is the injection surface for custom targets and the router-bypass boundary per D8; `SimulationResult` is the durable projection-artifact shape per D3 and D12. Accidental shape drift on any of them breaks every Node that runs simulations or consumes projections (Flow, Agents, Lore, HoneyHub when live). The canary makes this a compile-time failure at Sim's own CI, not a discovery at consumer sites. This matches the pattern ADR-0016 through ADR-0024 established of freezing the hot-path surfaces.

`Scenario` and `RiskAssessment` are not in the stand-up canary because their shapes are expected to evolve modestly as concrete `ISimulationTarget` shapes land at scaffold and surface per-scenario and per-risk-mode requirements. They become canary candidates once the scaffold packet lands and the shapes settle.

### New invariants (proposed for `constitution/invariants.md`)

Numbering is tentative — scope agent finalizes at acceptance.

- **Downstream Nodes take a runtime dependency only on `HoneyDrunk.Sim.Abstractions`.** Composition against `HoneyDrunk.Sim` and any `HoneyDrunk.Sim.Providers.*` package is a host-time concern. See D2.
- **Side-effect-freedom is the Sim–Evals boundary primitive.** If it writes to any Grid-durable surface, it is not Sim. Evals reads real behavior; Sim substitutes for real behavior to predict it. See D6.
- **Sim consumes `IWorkflow` and `IWorkflowStep` as read-only authoring surfaces, never `IWorkflowEngine`.** This is the reciprocal of ADR-0024 D4/D6's Flow-side pin. See D5.
- **Sim composes Operator primitives as observation-only prediction inputs.** `ISafetyFilter`, `ICostGuard`, and `IApprovalGate` are called to predict what Operator would do; Sim never enforces. See D9.
- **Router bypass is permitted only through `ISimulationTarget`.** Parallel to Evals's `IEvalTarget` bypass per ADR-0023 D6; `ISimulationTarget` does not share the `IEvalTarget` type (avoids a Sim → Evals runtime edge). Every bypass is recorded on the `SimulationResult`'s `ModelCapabilityDeclaration` provenance field per D12. See D8.
- **Every `SimulationResult` records the full run provenance — scenario identity and version, `ISimulationTarget` identity, `ModelCapabilityDeclaration` identity (when a model was pinned), reproducibility seed, timestamps.** `SimulationResult` without provenance is not a valid result. See D7 and D12.

- **Simulation telemetry is metadata-only and follows the Memory/Knowledge content-in-telemetry rule, not the Evals carve-out.** Scenario inputs and predicted outputs are not carried in Pulse signals; they are recoverable through `SimulationResult`. See D10.
- **`HoneyDrunk.Sim.Abstractions` does not take compile-time dependencies on `HoneyDrunk.Operator.Abstractions` or `HoneyDrunk.Memory.Abstractions`.** Both are runtime-only edges. See D2, D9, and D11.
- **The Sim Node CI must include a contract-shape canary for `ISimulator`, `IPlanValidator`, `ISimulationTarget`, and `SimulationResult`.** Shape drift on any of the four is a build failure, not a downstream discovery. See D13.

### Contract-shape canary becomes a requirement

The contract-shape canary in D13 is a gating requirement on the Sim Node's CI from the first scaffold. It is not a later hardening pass — the four frozen surfaces are the hot path for every Node that runs simulations or consumes projections and must be protected from day one.

### Negative

- **Six exposed surfaces (three interfaces plus three records) plus the router-bypass rule plus the side-effect-freedom primitive plus the fixture-composition rule is more public surface than a minimal "single `ISimulator` plus a projection bag" design would ship.** The trade is clarity of responsibility, a clean pinning primitive that does not widen AI's hottest interface, reproducible projections with full provenance, and independent testability against modestly more contract surface to version. Given the contract-shape canary on the four hot-path surfaces, the extra surface costs little to maintain.
- **Reconciling three drifted cataloged sources (contracts.json, relationships.json, repo overview) at the same time as standing up the Node is more follow-up work than a greenfield stand-up would require.** Accepted: the drift is the problem this ADR solves, and reconciling it now — rather than adding a fourth variant — is the point of pinning the D3 definitive set in this ADR. This matches the pattern ADR-0023 D3 applied to Evals's same three-way drift.
- **Shipping `Providers.InMemory` at stand-up without a production scenario-execution backend means no production-ready simulation substrate exists at first release.** Accepted: `InMemory` is enough to unblock all first-wave consumers on the contract surface, and the side-effect-freedom principle is pinned independent of the substrate per D6. The first production backend lands under its own issue packet once Flow or HoneyHub (when live) drives the shape.
- **The compile-time `HoneyDrunk.Sim.Abstractions` → `HoneyDrunk.Flow.Abstractions` edge means every downstream consumer of Sim inherits a transitive compile-time reference to Flow.** Accepted: Flow is a peer foundation Node, the reference is to `Abstractions` only (no runtime drag), and D5 names the edge explicitly as the reciprocal of ADR-0024 D4/D6. The alternative (Sim re-declaring its own workflow-definition shape) would produce drift and fork the workflow-definition surface — a higher cost than the compile-time transitive reference.
- **The parallel-but-distinct-types choice for `ISimulationTarget` versus Evals's `IEvalTarget` introduces two parallel target abstractions.** Accepted: Sim does not evaluate and Evals does not simulate — the Nodes do not compose — and sharing a type would create a Sim → Evals runtime edge neither Node needs. Parallel shapes with distinct types preserve the Node boundary and avoid coupling.
- **The observation-only composition of Operator primitives (D9) requires Sim runtime to compose three Operator surfaces that Sim never enforces against.** Accepted: the prediction semantics are structurally necessary — a plan validator that ignored cost, safety, and approval concerns would produce projections that are structurally useless — and the observes-vs-enforces framing (mirrored from ADR-0023 D7) keeps Sim from crossing into Operator's enforcement role.
- **Deferring Monte Carlo / N-trial distribution surfaces, concrete target shapes (`WorkflowTarget`, `AgentTarget`, `RetrievalTarget`, transcript-replay), and the production scenario-execution backend to scaffold means the first real consumer may surface needs that reshape these.** Accepted: this is the same cost ADR-0017 D6, ADR-0020 D12, ADR-0022 D11, ADR-0023 D13, and ADR-0024 D13 accepted for their equivalent deferrals, and it is cheaper than pinning a shape before a real consumer drives it.
- **Pulse ingress back into Sim (reactive scenario selection, reactive risk-threshold tuning) is deferred.** Consumer-driven scenario selection via App Configuration covers the stand-up need; automatic reactive tuning is a later concern and matches the emit-only stance every prior AI-sector stand-up took.

## Alternatives Considered

### Fold Sim into HoneyDrunk.Evals

Rejected per D6 and Q1 locked with the user. Evals consumes a target's real behavior to score it; Sim substitutes for a target's real behavior to predict what would happen. The primitive that separates the two is **side-effect-freedom** — if it writes to any Grid-durable surface, it is not Sim. Folding Sim into Evals would either (a) entangle Evals's real-run-observation charter with Sim's substitute-and-predict charter (breaking Evals's read-real-behavior rule per ADR-0023 D4), or (b) pull Sim's side-effect-freedom invariant into Evals's Node-local invariant set in a way that contradicts Evals's need to observe real writes during scoring. The two have different runtime contracts (Evals runs targets live; Sim never does), different state models (Evals produces durable `EvalReport` from real behavior; Sim produces `SimulationResult` from substituted behavior), and different downstream consumers. Keeping them separate preserves each Node's charter and lets downstream consumers choose between "did this actually happen well" and "would this happen well" without conflating the two.

### Fold Sim into HoneyDrunk.Flow

Rejected. Flow executes multi-step workflows live per ADR-0024 D1 — it runs them, persists state, compensates partial failures, and resumes after a pause. Sim dry-runs the same workflow shape against deterministic fixtures without any of those side effects. The two have different lifecycles (Flow runs real workflows; Sim never does), different state models (Flow writes durable `IWorkflowState`; Sim writes nothing durable from D6), and different downstream consumers. Folding them would either put simulation semantics into a live execution engine (breaking D6) or put live execution semantics into a simulation substrate (breaking the execute-vs-dry-run boundary `repos/HoneyDrunk.Sim/boundaries.md` already records). This is the reciprocal of ADR-0024 D14's "fold Flow into Sim" alternative — both directions are rejected for the same reason: the primitive that separates the two is the side-effect-freedom line, and collapsing the line produces a Node with contradictory invariants.

### Widen `IChatClient` for Sim router-bypass

Rejected per D8 Option A. Same reasoning as ADR-0023 D6 Option A. `IChatClient` is frozen under ADR-0016 D8's contract-shape canary; widening it for Sim's special case would inflate every caller's API surface and create a second model-selection path that competes with the routing layer. One Node's special case should not reshape the Grid's hottest interface.

### Sim composes `IModelProvider` directly

Rejected per D8 Option B. Same reasoning as ADR-0023 D6 Option B. Pulling `IModelProvider` composition into Sim's runtime would either duplicate the router's provider-discovery logic or introduce a second composition path that sees models the router does not see. Both produce drift, and both give Sim a routing-layer responsibility that belongs in AI.

### Sim reuses Evals's `IEvalTarget` directly

Rejected per D8 and Q2 locked with the user. A shared target type would require `HoneyDrunk.Sim.Abstractions` to take a compile-time dependency on `HoneyDrunk.Evals.Abstractions`, creating a Sim → Evals runtime edge neither Node benefits from — Sim does not evaluate and Evals does not simulate. The shapes are parallel but the types are distinct: `ISimulationTarget` for Sim, `IEvalTarget` for Evals. Each Node owns its own target abstraction. Sharing would couple two Nodes whose responsibilities do not overlap, for no compositional benefit.

### Keep `IScenario` / `IRiskAssessment` as interfaces

Rejected per D3 and Q4 locked with the user. `Scenario` and `RiskAssessment` carry value-type data — a scenario is initial state plus actions plus constraints (data), and a risk assessment is identified failure modes plus probabilities plus mitigations plus confidence (data). Both are record shapes, not behavior surfaces. The grid-wide naming rule applies: records drop the `I` prefix and are cataloged with `kind: "type"`; interfaces keep the `I` prefix and are cataloged with `kind: "interface"`. `ModelCapabilityDeclaration` (ADR-0016), `CapabilityDescriptor` (ADR-0017), the four governance records (ADR-0018), `KnowledgeSource` (ADR-0021), `EvalCase` and `EvalReport` (ADR-0023) set the precedent. Keeping `IScenario` / `IRiskAssessment` as interfaces would create a second convention in the catalog and make the naming rule unenforceable.

### Ship `HoneyDrunk.Sim.Testing` instead of `HoneyDrunk.Sim.Providers.InMemory`

Rejected. Sim has a clear provider-slot axis on the scenario-execution side (rule-based, model-backed via `IChatClient`, shared in-memory, replayed-transcript — any of which could ship later under the `Providers.*` family). The in-memory backend belongs on that axis as a production-shaped fixture under `Providers.InMemory`, not as a test-only `.Testing` artifact. The `.Testing` pattern ADR-0017 D2 set for Capabilities remains valid for that Node (where there is no provider-slot axis at the registry layer); it is the wrong shape for Sim, which matches Knowledge (ADR-0021 D2), Memory (ADR-0022 D2), Evals (ADR-0023 D2), and Flow (ADR-0024 D2) on this axis.

### Monte Carlo / N-trial distribution surface at stand-up

Deferred. A distribution surface — run the same scenario N times and surface a distribution of outcomes with percentiles and confidence intervals — is a legitimate simulation primitive for non-deterministic targets (chat-backed without a fixed seed, live-retrieval-under-simulation). At stand-up, the single-trial `ISimulator` shape in D3 plus the reproducibility seed in D7 plus the non-determinism flag on `ISimulationTarget` per D7 gives consumers enough to run distributions manually. The first-class distribution API shape lands at scaffold when the first real consumer drives it. Pinning the shape prematurely would likely require revision when HoneyHub (when live) or the first production plan-validator surfaces per-consumer requirements the ADR could not predict.

### Sim does not compose Operator primitives

Rejected per D9 and Q3 locked with the user. A plan validator that ignored cost, safety, and approval concerns would produce `RiskAssessment` projections that are structurally useless — a "plan looks fine" verdict that would be blocked by `ICostGuard` or `IApprovalGate` in a real run is the plan validator's failure. The observation-only composition pattern (mirrored from ADR-0023 D7) lets Sim predict what Operator would do without Sim enforcing — the observes-vs-enforces distinction is preserved, and policy prediction becomes part of plan evaluation. The Operator edge is added to Sim's `consumes` per the follow-up work section.

### Match Evals's content-in-telemetry carve-out

Rejected per D10. Evals's carve-out ADR-0023 D10 established (content permitted in eval signals unless the suite declares itself sensitive) is structurally motivated by regression diagnosis — an investigator reading a Pulse signal that "case 42 dropped from 0.95 to 0.62" needs the text of case 42's input and output to triage. Sim's telemetry does not face the same pressure. Simulation projections are forward predictions; the actionable information in a Pulse signal that "scenario X produced a risk score of 0.72" is the identity, the score, and the confidence — not the payload. Stripping content from simulation telemetry does not degrade actionability the way it would degrade eval triage. The conservative Memory/Knowledge rule (metadata-only, content recoverable through the durable artifact) applies. If a specific use case surfaces later that argues for the carve-out, the right path is a scaffold-time opt-in flag analogous to Evals's sensitivity flag, not a stand-up-time carve-out.

### Pulse signal ingress into Sim at stand-up

Deferred per D10. Reactive closed-loop tuning — where observed Pulse telemetry automatically re-selects scenarios, adjusts risk thresholds, or triggers re-runs — is the same class of concern ADR-0018 flagged for Operator, ADR-0020 flagged for Agents, ADR-0021 flagged for Knowledge, ADR-0022 flagged for Memory, ADR-0023 flagged for Evals, and ADR-0024 flagged for Flow. It is not a stand-up decision. Emit-only at stand-up is the committed direction; any future ingress contract will be added as a distinct ADR.

### Defer the Sim stand-up until HoneyHub needs it

Rejected. Flow's reciprocal edge to Sim is already pinned at ADR-0024 D4/D6 ("Sim consumes `IWorkflow` as read-only"), and Lore, Agents, and HoneyHub (when live) each list Sim as a planned dependency. Deferring Sim would leave the reciprocal edge half-pinned (Flow side committed, Sim side absent) and would push every downstream consumer into either blocking on Sim or inventing its own simulator — the divergence across those local simulators would be immediate. Standing up Sim now **closes the AI-sector stand-up wave**: every cataloged AI-sector Node has a stand-up ADR, the foundation is shaped, and downstream Nodes can compose AI-sector substrates against a stable, fully-pinned contract surface.
