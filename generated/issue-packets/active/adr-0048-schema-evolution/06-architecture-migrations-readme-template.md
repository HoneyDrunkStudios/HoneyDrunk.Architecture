# Author the per-Node Migrations/README.md template

## Summary
Author a canonical `Migrations/README.md` template that every migration-bearing Node copies into its `src/HoneyDrunk.<Node>.Data/Migrations/` folder. The template records the per-Node migration history overview, naming conventions, the Expand → Migrate code → Contract phase pairing log, online-DDL caveats per ADR-0048 D6, the `[Rollback]` declaration convention, and a pointer to the `migrate.yml` workflow caller. The template lives in the Architecture repo as a reference doc; each Node copies it into its own repo at standup or first migration.

## Context
ADR-0048's Follow-up Work names: "Author the `Migrations/README.md` template for per-Node repos."

ADR-0048 D11 specifies the file's home: "`src/HoneyDrunk.<Node>.Data/Migrations/README.md` — per-Node migration history overview + caveats." The template is the boilerplate every Node fills in.

**Why the template lives in the Architecture repo (not in each Node).** A single canonical source means: (1) when ADR-0048's policy evolves, the template updates in one place; (2) the `database` review agent (packet 02) can reference the template structure when checking per-Node `Migrations/README.md` for conformance; (3) Nodes that haven't yet adopted migrations (Audit, Memory, Knowledge, Billing — all Seed or pre-existing-without-migrations) have a definite starting point at standup. The per-Node copy is the *instance*; the template is the *type*.

**Discoverability.** The template lives in the Architecture repo at a stable path. Reasonable candidates:
- `templates/per-node/Migrations-README.md` (a new top-level `templates/` directory, with sub-folders for per-Node templates).
- `repos/_template/Migrations-README.md` (under the existing `repos/` directory tree, with `_template/` as the per-Node skeleton).
- `docs/migrations/per-node-readme-template.md` (under a `docs/` tree).

Pick whichever fits the repo's existing convention; favour the existing `templates/` or per-Node-template location if one exists. If no convention is established, create `templates/per-node/Migrations-README.md` and note the path in `repos/README.md` or a similar index file so Node maintainers know where to find it.

This is a docs/template packet. No code, no .NET project.

## Proposed Implementation
1. **Author the template.** Sections, in order:

   ```markdown
   # Migrations — {Node Name}

   This Node's relational store is evolved via EF Core Migrations per ADR-0048 D1. Document-store evolution (if any) follows schema-on-read per ADR-0048 D7 — see the Schema-on-read section below.

   ## Framework

   - **EF Core version:** {e.g. 9.x} — tracks the Grid's LTS alignment per ADR-0034.
   - **Schema-evolution posture:** `ef-core-migrations` (matches `catalogs/grid-health.json` `schema_evolution` field for this Node).
   - **Provider:** Azure SQL | Postgres.

   ## File layout

   ```
   src/HoneyDrunk.<Node>.Data/
     Migrations/
       <YYYYMMDDHHmmss>_<DescriptiveName>.cs         # generated by `dotnet ef migrations add`
       <YYYYMMDDHHmmss>_<DescriptiveName>.Designer.cs
       <Node>DbContextModelSnapshot.cs
       README.md                                      # this file
     Migrations/Backfill/
       <YYYYMMDD>-<description>.md                    # document-store backfill runbooks per ADR-0048 D7
   ```

   ## Naming convention

   - Migration class name: `<Verb><Subject>` PascalCase, no spaces. Examples: `AddTenantTierColumn`, `BackfillAuditEmitterIndex`, `DropLegacyCorrelationId`.
   - File name: EF Core's default `<timestamp>_<ClassName>.cs` pattern.
   - Phase annotation: every migration class that is part of an Expand/Contract pair carries `[ExpandPhase("MIG-YYYY-MM-DD-NNN-<short-id>")]` or `[ContractPhase("MIG-YYYY-MM-DD-NNN-<short-id>")]` per ADR-0048 D5. The `<short-id>` is shared across the pair.
   - Tier 2 breaking change: `[BreakingChange("reason")]` per ADR-0048 D2 (allowed only on Tier 2 internal stores).
   - Rollback declaration: every migration class carries `[Rollback(Strategy = RollbackStrategy.ForwardMigration, Notes = "...")]` (default) or `[Rollback(Strategy = RollbackStrategy.NonRollback, Reason = "...")]` per ADR-0048 D10/D12.

   ## Expand → Migrate code → Contract phase log

   | Migration ID | Phase | Migration file | Code-PR ref | Contract phase ref | Notes |
   |--------------|-------|----------------|-------------|--------------------|-------|
   | MIG-YYYY-MM-DD-NNN-<short-id> | Expand | <timestamp>_<ClassName>.cs | #PR | #PR | {what the Expand adds; old shape retained} |
   | MIG-YYYY-MM-DD-NNN-<short-id> | Contract | <timestamp>_<ClassName>.cs | #PR | — | {what the Contract drops; old shape removed} |

   Audit Node only: **Contract phase is always "N/A — append-only-by-interface per ADR-0030 D4 / ADR-0048 D8."**

   ## Backward-compatibility window

   - **Audit:** indefinite (≥ 730-day retention; no Contract phase per ADR-0048 D8).
   - **Tier 0 / Tier 1 customer-facing stores (Notify Cloud, Memory, Knowledge, Billing):** 2 stable deploys minimum or 14 days, whichever is longer, between Expand and Contract.
   - **Tier 2 internal stores (Pulse historical, Evals):** 1 stable deploy minimum between Expand and Contract.
   - **Idempotency stores:** 2 stable deploys minimum or 30 days (matching ADR-0042 D4 billing/audit TTL).
   - **Vault and adjacent secret-state stores:** N/A — Key Vault has no relational schema; secret-shape evolution is governed by ADR-0006, not ADR-0048.

   ## Online DDL caveats (ADR-0048 D6)

   For tables ≥ 100k rows, online DDL primitives are required:

   - **Postgres:** `CREATE INDEX CONCURRENTLY`, `ALTER TABLE ... VALIDATE CONSTRAINT` after `ADD CONSTRAINT ... NOT VALID`.
   - **Azure SQL:** `CREATE INDEX ... WITH (ONLINE = ON)`, `ALTER TABLE ... WITH CHECK CHECK CONSTRAINT` after `WITH NOCHECK ADD CONSTRAINT`.

   Use `MigrationBuilder.Sql(...)` for the SQL escape hatch where the fluent API doesn't emit the desired DDL.

   ## Audit-specific constraints (ADR-0048 D8) — Audit Node only

   - No column drops, ever.
   - No type narrowing, ever.
   - No `NOT NULL` additions to existing columns.
   - New columns are nullable, always.
   - Paired-table pattern for breaking changes: `AuditEntryV2` alongside `AuditEntry`, queries UNION across.
   - Indexes may be added or dropped (does not affect row data).

   ## Schema-on-read (ADR-0048 D7) — for any document-store sub-piece of this Node

   If this Node ships a Cosmos-backed or document-store-backed sub-piece (e.g. the Kernel idempotency dedup store):

   - **New fields:** added by writing them; no DDL.
   - **Reading code:** tolerates absence; missing fields default to type zero values.
   - **Backfill:** an operator-triggered job, documented as `Migrations/Backfill/<YYYYMMDD>-<description>.md`. NOT an EF migration.
   - **Field removal:** mirrors relational Contract phase — only after all reading code has dropped the reference, and only after a window matching D5 for the store's tier.
   - **Partition-key changes:** NOT possible in place. A container migration — new container, dual-write, backfill, dual-read, cutover, drain, delete. Triggers a follow-up ADR per ADR-0048 D7.

   ## Running migrations

   Migrations run via the `migrate.yml` reusable workflow in HoneyDrunk.Actions per ADR-0048 D3 and D11. They never run at app startup, never via init container (invariant 94 from ADR-0048).

   To migrate this Node's schema:

   1. The migration PR lands first; the migration assembly builds into the Node's CI artifacts; no production migration runs yet.
   2. The operator triggers the per-environment caller workflow (e.g. `.github/workflows/migrate-dev.yml`), which calls `migrate.yml` with `(node, environment, db-provider)` inputs.
   3. The operator triggers the dependent code deploy per ADR-0033 D1 (push for `dev`; tag for `staging`/`prod`).
   4. After the deploy is stable and the old revision is drained, a future PR (Contract phase, if applicable) lands and is migrated through the same flow.

   ## Failure recovery

   See `dr-runbook.md` Migration Failure section (per ADR-0036 D9 + ADR-0048 D10) for the per-Node failure recovery procedure.

   ## Tests

   Every migration carries a Tier 2b round-trip test in `tests/HoneyDrunk.<Node>.Tests.Integration.Containers/Migrations/<Node>MigrationRoundTripTests.cs` per ADR-0048 D12 and ADR-0047 D11. The test applies all migrations against a Testcontainers backing, inserts representative data, reads it back, and asserts integrity. For Expand-phase migrations, it also tests forward-compatibility (old-shape data + new code). For Contract-phase migrations, it asserts the old shape is gone and any old-shape data has migrated to the new shape.

   ## See also

   - ADR-0048 — Data Schema Evolution and Migration Policy (the policy this README implements).
   - ADR-0030 — Grid-Wide Audit Substrate (the append-only-by-interface invariant Audit follows).
   - ADR-0042 — Idempotency Contract for Async Boundaries (the idempotency-store TTL and the Cosmos schema-on-read backing).
   - ADR-0047 — Testing Patterns and Tooling (Tier 2b round-trip test placement).
   - ADR-0033 — Environment-Gated Deploy Trigger Model.
   - `migrate.yml` reusable workflow — `HoneyDrunkStudios/HoneyDrunk.Actions/.github/workflows/migrate.yml`.
   ```

**ADR-0048 D2 — Expand → Migrate code → Contract pattern.** The template's Expand/Contract phase log is the per-Node record of which migrations are part of which pair.

**ADR-0048 D7 — Document stores schema-on-read.** New fields by writing; reading code tolerates absence; backfill via `Migrations/Backfill/<YYYYMMDD>-<description>.md`; partition-key changes are container migrations and a deferred-ADR trigger.

**ADR-0048 D8 — Audit table specifics.** No column drops, no type narrowing, no `NOT NULL` add, new columns nullable, paired-table pattern for breaking changes.

**ADR-0048 D10 — Forward-only rollback; `[Rollback]` attribute on every migration.**

**ADR-0048 D12 — Round-trip test per migration in `HoneyDrunk.<Node>.Tests.Integration.Containers/Migrations/`.**

**ADR-0048 D14 — Phased rollout.** Phase 1 (Week 1-2) is the foundation work this packet ships; subsequent phases adopt the template per Node.

## Constraints
- **The template is reference content, not a per-Node action.** This packet does NOT edit any Node repo's `Migrations/README.md`. Per-Node adoption follows ADR-0048 D14's phases — Notify retroactively (packet 07), Kernel.Idempotency at Phase 2 (packet 08), Audit at standup (deferred to ADR-0031), Memory/Knowledge/Billing at their standups (deferred).
- **Pick the template path per existing convention.** Don't invent a new directory tree if one fits.
- **All per-tier windows match ADR-0048 D5 verbatim.** Do not paraphrase or round (e.g. don't write "~14 days"; write "14 days").
- **Audit-specific constraints are flagged for Audit only.** The template applies to all migration-bearing Nodes; the Audit section is conditional ("Audit Node only").
- **No `[Rollback]` attribute behavior beyond what D10 specifies.** The template documents the attribute's two strategies (`ForwardMigration` default, `NonRollback` with required `Reason`). It does NOT invent additional strategies.
- **Cite ADR-0048 D-decisions inline.** The template is read by Node maintainers who may not have the full ADR loaded; inline citations make each rule traceable to the source decision.
