# Wire queue-depth surfacing into hive-sync and the weekly ADR-0043 briefing

## Summary
Wire two surfaces to make worker availability observable as an advisory signal per ADR-0086 D7: the weekly ADR-0043 briefing surfaces PRs in `needs-agent-review` older than 24 h as a backlog signal, and `hive-sync` (per ADR-0014's reconciliation mandate) reports current queue depth as a Grid-health metric. Both surfaces consume the same data (the `needs-agent-review` label count Grid-wide). No new pager, no inbound alert — the queue itself is the signal.

## Context
ADR-0086 D7 makes worker availability explicitly **advisory** like agent verdicts. Worker downtime — operator traveling, home server down, scheduled task disabled, deliberate maintenance — produces PRs that accumulate in `needs-agent-review` and **remain mergeable**. The advisory posture from ADR-0011 D5 (preserved by ADR-0044 D10, preserved here) explicitly accepts this. The signal model:

- **GitHub mobile notifications** on label-state changes and PR comments — operator already gets these for free.
- **Weekly ADR-0043 briefing** surfaces PRs in `needs-agent-review` older than 24 h as a backlog signal — "earns its keep because a backlog older than 24 h is exactly the case where the operator wants to know the worker is down."
- **`hive-sync` / `netrunner` queue-depth reporting** flags queue depth as a Grid-health metric per ADR-0014's reconciliation mandate.

The first surface (GitHub mobile) is already configured by the operator and needs no work. This packet wires the second and third surfaces.

The work is small: identify where the ADR-0043 weekly briefing pulls its inputs and where `hive-sync` writes Grid-health metrics, then add a single field/section to each. Both inputs are a single `gh api search/issues` call equivalent to the worker's tick poll (one call to count PRs across the Grid with `needs-agent-review`, optionally filtered by age).

## Scope
- Identify the source of truth for the ADR-0043 weekly briefing inputs (per `.claude/agents/` or wherever the briefing-generation agent reads inputs from) and add a "Grid Review Runner queue" section listing PRs in `needs-agent-review` older than 24 h.
- Identify how `hive-sync` (per ADR-0014 Phase 6 drift report, or wherever the queue-depth metric lands) reports Grid-health metrics and add a `grid_review_queue` field with `total` (current `needs-agent-review` count Grid-wide) and `stale_count` (count older than 24 h).
- Document the surfacing in `.claude/agents/hive-sync.md` (or the canonical hive-sync agent definition) and `.claude/agents/` whichever runs the weekly briefing (likely the same or a sibling).

## Proposed Implementation

### Identify the briefing input source
The weekly ADR-0043 briefing is generated by a recurring agent run (per ADR-0043). The agent's prompt enumerates the inputs it consumes — Strategic/Tactical/Reactive sources. Add a new "Worker queue" input section to the briefing agent's prompt:

```markdown
## Worker queue (ADR-0086 D7)

Pull PRs across the HoneyDrunkStudios org carrying the `needs-agent-review` label
that were created (or last labeled) more than 24 hours ago. Surface them in the
briefing under a "Stalled in Grid Review Runner queue" subsection of the
"Reactive" pillar. The default expectation is that this list is empty; a
non-empty list means the worker has been offline > 24 h (per ADR-0086 D7's
backlog signal).

Source: `gh api -X GET search/issues -f q='is:pr is:open label:needs-agent-review org:HoneyDrunkStudios updated:<{since}'`
where `{since}` is `now() - 24h` in ISO-8601 UTC.

If the list is empty, the briefing notes "Worker queue: clean."
If non-empty, the briefing names each PR with link, age, and the worker host
identifier from its queue comment (so the operator can see which worker
should have processed it).
```

The exact agent file edit depends on the briefing agent's current file location (likely `.claude/agents/brief.md` or similar — the implementing agent locates it via the `.claude/agents/` directory listing and the ADR-0043 references).

### Wire hive-sync queue-depth metric
`hive-sync` (per ADR-0014 Phase 6's drift-report) writes Grid-health metrics. Identify where the drift report or the periodic hive-sync output lands a `grid_health` JSON (or equivalent). Add a `grid_review_queue` block:

```json
{
  "grid_review_queue": {
    "total": 3,
    "stale_over_24h": 0,
    "stale_over_72h": 0,
    "as_of": "2026-05-27T14:32:00Z",
    "worker_recent_tick": "2026-05-27T14:31:42Z"
  }
}
```

`worker_recent_tick` is optional — populated by the worker writing a heartbeat file or by reading the most-recent claim comment's `claimed_at` Grid-wide. If too heavy, drop it; the `total` + `stale_over_24h` + `as_of` triple is the minimum viable signal.

Update `.claude/agents/hive-sync.md` (the canonical agent definition) to document the new field and its meaning. Per ADR-0014, hive-sync is the canonical reconciliation agent; this is a tidy place for the queue-depth field to live.

### Documentation

- `.claude/agents/brief.md` (or wherever the ADR-0043 briefing agent lives) — document the new "Stalled in Grid Review Runner queue" subsection.
- `CHANGELOG.md` — entry noting the wiring.

### What this packet does NOT do
- Does **not** build a separate dashboard, alarm, or Telegram/Discord notifier. ADR-0086 D7 is explicit: no pager, no alarm, no inbound channel. The queue itself is the signal.
- Does **not** change the worker source. The worker already logs each tick's depth (per packet 03 acceptance criteria); this packet only surfaces the existing data through the two narrative surfaces.
