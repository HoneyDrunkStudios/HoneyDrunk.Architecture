# Phase 2 — grant `docs-sync` cross-repo PR authority, auto-fix scoped to version drift only

## Summary
Activate Phase 2 of the ADR-0085 rollout: update the active-phase guard in `.claude/agents/docs-sync.md` to permit `gh pr create` against target repos, but constrain the auto-fix scope to **D3 category 2 (version drift) only**. All other detection categories remain report-only at this phase. Document the PR-metadata discipline (Authorship + OOB-reason), the branch-naming convention (`chore/docs-sync-{YYYY-MM-DD}`), the per-repo branch-reuse rule from D7, the fallback `proposed/` packet path for editorial findings, and the three-consecutive-failed-runs backoff. This is the **first packet that grants cross-repo write authority** — Phase 1 was report-only.

## Context
ADR-0085 D8 Phase 2 says: "Cross-repo PR authority enabled for mechanical fixes. Agent gains `gh pr create` against target repos. Auto-fix scope at this phase: **version drift (#2) only** — the highest-signal, lowest-judgment category. Every other finding remains report-only. One PR per affected repo per run, with the `Authorship: agent-claude-code` + OOB-reason metadata."

The reason Phase 2 caps at version drift only: version-drift fixes are mechanically resolvable by exact-string comparison (`<Version>` in `*.csproj` vs. README/CHANGELOG/install-snippet text), idempotent (re-running produces no diff), and have the lowest false-positive rate of all six categories. Phases 3–5 broaden the scope only after Phase 2's PR pipeline is observed working.

This packet is docs/governance/agent-prompt-update only. No .NET code, no GitHub Actions workflow change, no new infrastructure. `Actor=Agent`.

## Proposed Implementation

### Active-phase guard update
The Phase-1 guard rejected any `gh pr create` against a target repo. The Phase-2 update:

- Permits `gh pr create` against target repos, but only when the finding category is `version-drift` (D3 #2).
- Continues to refuse `gh pr create` for findings in categories 1, 3, 4, 5, 6 — those remain report-only at Phase 2 (with the editorial-fallback packet path for categories 3, 5, 6 and an unresolved-link disposition for category 4, which becomes auto-fixable in Phase 3).
- The guard is the runtime enforcement of D8's per-phase scope.

Concretely, the guard reads something like:

```
ACTIVE PHASE: 2 (Phase 2 per ADR-0085 D8 — cross-repo PRs enabled, scoped to version drift only)
PERMITTED AUTO-FIX CATEGORIES: ["version-drift"]
REPORT-ONLY CATEGORIES (Phase 2): ["missing-required-artifact", "symbol-reference-drift",
                                   "catalog-reference-drift", "dependency-graph-drift",
                                   "agent-instruction-drift"]
FALLBACK PACKET PATH (Phase 2): ["symbol-reference-drift", "dependency-graph-drift",
                                 "agent-instruction-drift"] — write to
                                 generated/issue-packets/proposed/{YYYY-MM-DD}-{repo}-docs-{slug}.md
```

If a future packet (03–05) widens the auto-fix scope, the guard wording is the single edit point.

### PR-metadata discipline (per ADR-0085 D4, ADR-0044 D6)
Every cross-repo PR opened by the agent must satisfy `pr-core.yml` Job 7 (Authorship Check) and Job 8 (PR Metadata Check). The required body block:

```
Authorship: agent-claude-code
Out-of-band reason: Generated by docs-sync run {YYYY-MM-DD}; full report at
HoneyDrunk.Architecture/generated/docs-sync-reports/{YYYY-MM-DD}.md
```

- `agent-claude-code` is the existing enum value used because Claude Code is the execution surface. Adding `agent-docs-sync` to the enum is deferred to Phase 6 (packet 06) per ADR-0085 D4.
- The `Out-of-band reason:` points at the per-run report path. This is operationally equivalent to a packet link, modeled on `hive-sync`'s reconciliation PRs which follow the same pattern.
- The `out-of-band` label is auto-applied by the pr-core OOB pattern.

### Branch naming + reuse rule (per D7)
- Branch name: `chore/docs-sync-{YYYY-MM-DD}`.
- If a PR with that branch already exists and is **open** at the start of the next run, the agent **reuses the branch** and pushes additional commits rather than opening a parallel PR. This honors the "one PR per repo per initiative" operator convention and avoids PR hell.
- If the prior week's PR was merged or closed, the new run opens a fresh PR with the new date in the branch name.

### Fallback packet path (per D4)
Even in Phase 2, the agent must be able to write a `proposed/` packet when an editorial finding surfaces (categories 3, 5, 6). The packet:

- Lives at `generated/issue-packets/proposed/{YYYY-MM-DD}-{repo}-docs-{slug}.md`.
- Follows the `copilot/issue-authoring-rules.md` format.
- Frontmatter: `source: reactive`, `generator: docs-sync`, `target_repo: HoneyDrunkStudios/HoneyDrunk.{Repo}`, `wave:` omitted (standalone-shaped), `initiative:` omitted, `node:` set to the target Node.
- Subject to the **editorial-finding packet dedup rule** (D7): before writing a new packet for a Node + finding category, the agent checks for an existing un-triaged packet with `generator: docs-sync` covering the same Node and the same category. If found, the existing packet is left alone (invariant 24 protects packets from agent edits post-creation) and a one-line note is added to the run report: "Skipped — existing `proposed/` packet `{path}` still pending triage."

### PR-failure backoff (per D7)
If three consecutive weekly runs against the same repo:
1. open or update a PR (i.e., the run produced findings worth auto-fixing for that repo), and
2. that PR fails `pr-core`, and
3. is not merged or closed by the human within the week,

then on the third consecutive failure the agent:
- stops auto-pushing to that repo's PR (the existing PR is left open for human action),
- converts all that repo's findings to `proposed/` packets (subject to the dedup rule above),
- alerts the operator in the run report's "Stale Findings" or new "PR-failure backoff" section.

This prevents zombie-PR loops while preserving the operator's ability to intervene.

### Version Drift Detection — file-set + line-pattern specification

The version-drift detector scans a precisely-scoped file set against a precisely-scoped source of truth, with explicit ignore rules. This spec is load-bearing: any drift in the spec is a drift in what "version drift" means, and Phase 2's auto-fix authority depends on the spec being unambiguous.

**Files scanned (per-project, not just repo root):**
- For each `*.csproj` in the repo, the per-project README at `{project-directory}/README.md`. The Grid convention is that each public NuGet package has a README colocated with its `.csproj`; the repo-root `README.md` is the Node-level overview and is NOT the version-drift target.
- Within each per-project README, only the "Install" / "Usage" / "Getting Started" section (whatever the per-package README's snippet section is called by the existing template).

**Line patterns matched (in the Install section of each per-project README):**
1. `dotnet add package HoneyDrunk.X --version N.N.N`
2. `nuget install HoneyDrunk.X -Version N.N.N`
3. `<PackageReference Include="HoneyDrunk.X" Version="N.N.N" />`

Each pattern's `N.N.N` (or `N.N.N-suffix` for pre-release) is compared to the `<Version>` element in the corresponding `{project-directory}/*.csproj`. The MSBuild-resolved version (after `Directory.Build.props` interpolation) is the source of truth; the per-project README's snippet is the rewrite target.

**Files explicitly NOT scanned:**
- `CHANGELOG.md` — historical version rows by definition. Only the unreleased / current section of CHANGELOG would be relevant, and CHANGELOG is the auto-generated history of `<Version>` over time; rewriting historical CHANGELOG rows would be revisionist. Scope it out entirely at v1.
- Repo-root `README.md` — Node-level overview; if it cites a package version (rare), the agent treats it as report-only at Phase 2 (auto-fix may broaden in a later phase or never).
- Any file outside `{project-directory}/` for that project's snippet — cross-project version citations are out of scope (they fall under catalog-reference drift, Phase 3).

**Rewrite scope:**
- Only the matched line in the matched section of the matched per-project README is rewritten. The rewrite replaces the version segment with the `<Version>` element's resolved value. The replacement is exact-string for the version portion only; surrounding text (the rest of the line, the section heading, other paragraphs) is preserved verbatim.

**Idempotency:** Re-running the version-drift fix after the previous run merged produces no diff. The agent verifies this on read-back before considering the PR commit complete.

## Constraints
> **Invariant 24:** Issue packets are immutable once filed as a GitHub Issue. (Applied to the fallback packet path: existing un-triaged `proposed/` packets are not edited by the agent on subsequent runs — the dedup rule skips them and notes the skip in the run report.)

> **Invariant 25:** Dispatch plans are initiative narratives, not live state. The org Project board is the source of truth for in-flight work. (Applied here as: docs-sync's cross-repo PRs and fallback `proposed/` packets are listed in the run report — a per-run narrative — not in any live tracker.)

> **ADR-0043 D3 (verbatim):** Agents never self-promote `proposed/` → `active/`. (Preserved here: docs-sync's fallback packets land in `proposed/` and wait for human triage at the weekly briefing per ADR-0043 D5. The cross-repo PRs are a different surface — mechanical reconciliation rather than packet promotion — and the distinction is the one ADR-0085 D4 makes load-bearing.)

> **ADR-0085 D4 (verbatim qualifier):** Mechanical, exact-string drift is auto-fixed in the PR; anything requiring editorial judgment is surfaced for human action via TODO blocks in the PR body or a fallback `proposed/` packet. (The Phase-2 active-phase guard enforces this for version drift; later phases will reach the other mechanical categories.)

- **Active-phase guard is mandatory.** Without it, the agent could act beyond Phase 2 scope. The guard is the single edit point for future phase activations (packets 03–05 each bump it by one phase).
- **One PR per repo per run.** If the prior week's PR is open and unmerged, reuse the branch — never open a parallel PR for the same repo in the same initiative.
- **Editorial findings dedup before they become `proposed/` packets.** Without dedup, the agent will spam `proposed/` weekly with the same finding while the human hasn't triaged it yet.
- **The agent does not merge its own PRs.** ADR-0085 Alternatives Considered explicitly rejects this. The human is the PR-review gate; the agent opens and updates PRs.
- **PR metadata for this packet's implementation PR:** `Authorship: agent-claude-code` + `Packet: HoneyDrunkStudios/HoneyDrunk.Architecture#<issue-number>` once filed.
