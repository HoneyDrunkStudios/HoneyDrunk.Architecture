# CI Change: `nightly-deps.yml` opens/updates a single grouped `📦 Outdated Dependencies` issue per repo

## Summary
Extend `nightly-deps.yml` to open or update **exactly one** grouped per-repo tracking issue titled `📦 Outdated Dependencies` enumerating all outdated packages (`PackageName: current → latest`), found-or-created by stable title, body fully replaced in place every run, auto-closed when the outdated set is empty and reopened/recreated when packages fall behind again — the same idempotent find-or-update-by-stable-title pattern the `🕸️ Grid Health` issue uses.

## Target Repo
`HoneyDrunkStudios/HoneyDrunk.Actions`

## Motivation
Per the Grid's package-scanning posture (runtime packet-data id `adr-0009`), `nightly-deps.yml` runs weekly, reports outdated packages as artifacts and a step summary, and does **not** create per-package PRs (`create-update-prs` stays `false`, no Dependabot). But the report lives only in workflow artifacts (30-day retention) and a run summary — there is no durable, in-place, per-repo surface a human or agent can look at to answer "what's behind in this repo right now." A solo developer with eleven-plus repos and dozens of dependencies each would drown in per-package issues; the answer is **one grouped issue per repo, updated in place by stable title** — the minimum-noise surface, analogous to how the grid-health aggregator made pipeline health centrally visible without per-failure noise.

## Architecture decisions this packet implements (executor has no ADR access)

- **Exactly one issue per repo.** Title is the stable literal `📦 Outdated Dependencies`. **Not** one issue per package. **Not** one PR per package. Found-or-created by exact title match.
- **Body fully replaced in place on every run.** No appending; the issue body is always the current full outdated set.
- **Auto-close when empty; reopen/recreate when non-empty again.** When the outdated set becomes empty, the issue is closed automatically. When packages fall behind again on a later run, the issue under the same title is reopened (if it still exists, closed) or recreated (if it was deleted).
- **The issue lives in the repo whose packages are outdated** (the code lives where the work lives), created/updated by `nightly-deps.yml` running in that repo, using the `issues: write` caller permission already part of the canonical `nightly-deps.yml` caller `permissions:` block.
- **Reuse the existing `deps/report-dotnet` / `deps/consolidate-reports` composite actions** for the package data — do not re-implement `dotnet list package --outdated` parsing. The `consolidate-reports` action already produces `consolidated-dependency-report.json`; that is the input for the issue body.
- **`create-update-prs` stays `false`; no Dependabot; no per-package PRs.** Unchanged by this packet — this packet adds only the grouped-issue write responsibility on top of the existing report/artifact output.

## Proposed Implementation

### `nightly-deps.yml` — extend the `consolidate-report` job (or add a sibling job)

The existing `consolidate-report` job already runs `deps/consolidate-reports` to produce `./consolidated-dependency-report.json` and writes a step summary. Add to that job (after consolidation) a step `Maintain grouped tracking issue`:

1. **Add `issues: write` to the job's permissions.** The job currently inherits the workflow-level `permissions: contents: read`. The grouped-issue write needs `issues: write`. Add a job-scoped `permissions:` block:
   ```yaml
   permissions:
     contents: read
     issues: write
   ```
   (The canonical caller `permissions:` block for `nightly-deps.yml` callers already includes `issues: write` per the established CI/CD control-plane caller-permissions guidance; this job-scoped block is what consumes it.)

2. **Parse `consolidated-dependency-report.json`** into a sorted list of `PackageName: current → latest` lines. Group by project where the report carries project attribution (the `consolidate-reports` output includes per-report provenance; render as `#### <project>` subsections when more than one project has outdated packages, otherwise a flat list). Exclude prerelease per the existing `exclude-prerelease` input (already honored by `report-dotnet`).

3. **Render the issue body** (markdown):
   - A one-line header: `_Last scanned: <ISO-8601 UTC> · <N> package(s) behind · run: <run url>_`
   - The grouped `PackageName: current → latest` list.
   - A literal footer sentence: `Non-blocking. Outdated ≠ vulnerable — vulnerable packages block PRs separately. This issue is regenerated by nightly-deps every run; do not edit by hand.`
   - **No ADR number in the body.** Do not write the ADR id into the issue body prose.

4. **Find-or-update-by-stable-title** (the idempotent pattern):
   - `gh issue list --repo <this repo> --search 'in:title "📦 Outdated Dependencies"' --state all --json number,title,state` and filter to the **exact** title match `📦 Outdated Dependencies` (reject substring/variation-selector-normalized partials, exactly as the grid-health aggregator does for its emoji title).
   - **If outdated set is non-empty:**
     - Found and open → `gh issue edit <n> --body-file <rendered>` (full replace).
     - Found and closed → reopen (`gh issue reopen <n>`) then edit body.
     - Not found → `gh issue create --title "📦 Outdated Dependencies" --body-file <rendered>` and apply a `dependencies` label (create the label inline if it does not exist).
   - **If outdated set is empty:**
     - Found and open → edit body to a short "✅ All dependencies current as of `<ts>`." line, then `gh issue close <n> --comment "All dependencies current as of <ts> UTC (run <url>)."`
     - Found and closed → no-op.
     - Not found → no-op (nothing to track, nothing to create).

5. **Emoji-title idempotency caution.** `📦` is `U+1F4E6` (no variation selector — simpler than the spider-web case), but apply the same defensive exact-title filter and the same one-shot pre-flight verification recommended for the grid-health title (see Human Prerequisites). If `gh issue list --search` proves unreliable for this title in this org, fall back to caching the issue number in a repo Actions Variable `OUTDATED_DEPS_ISSUE_NUMBER` (read on each run; write back on first creation) exactly as the grid-health aggregator's documented fallback.

6. **The whole step must be non-blocking (D4 alignment).** This job already has `if: always()`. The issue-maintenance step must additionally be wrapped so a `gh` API hiccup logs a warning and exits 0 — `nightly-deps.yml` must never fail the run on issue-maintenance trouble. (The workflow is already declared "does NOT block merges or fail builds"; preserve that.)

7. **Same-repo token suffices.** The issue lives in the repo running `nightly-deps.yml`, so the default `GITHUB_TOKEN` with `issues: write` is sufficient. No cross-repo PAT is needed (unlike the grid-health aggregator, which writes into *other* repos).

### `docs/consumer-usage.md` update

Document that consumers' `nightly-deps.yml` caller must grant `issues: write` (most already do per the canonical caller `permissions:` block) and that a single `📦 Outdated Dependencies` issue per repo is now maintained automatically — do not hand-create or hand-edit it.
