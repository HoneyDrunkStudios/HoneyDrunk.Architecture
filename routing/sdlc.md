# SDLC — Development Lifecycle

How work moves from intent to execution across the HoneyDrunk Grid. Three AI surfaces handle different phases of the lifecycle. The developer operates across all three but spends most time in the planning surface.

---

## The Three Surfaces

### 1. Claude Code — Plan, Orchestrate, Hand Off

**Where:** Terminal / desktop, operating against the Architecture repo and workspace
**Role:** The brain. Thinks across repos, decomposes work, generates issues and handoff packets, makes architectural decisions.

**Responsibilities:**
- Goal decomposition → Features → Tasks
- Cross-repo impact analysis using catalogs and relationships
- Work item generation (structured GitHub Issues)
- ADR drafting when changes cross boundaries
- Codex handoff generation (task prompt + context bundle)
- Signal interpretation and plan adjustment
- Architecture discussions and design decisions

**When to use:**
- Starting any new initiative or feature
- Work that spans multiple repos
- Anything that requires understanding the Grid topology
- Generating work for Codex to execute
- Reviewing and adjusting plans based on execution results

**Does not:** Write production code. Open PRs. Deploy.

---

### 2. Codex — Execute in the Cloud

**Where:** Cloud-based, triggered by GitHub Issues or handoff prompts
**Role:** The hands. Takes a well-scoped task, executes it autonomously, opens a PR.

**Responsibilities:**
- Implement features scoped to a single repo
- Write tests for new or existing behavior
- Apply refactors described in work items
- Version bumps, changelog updates, dependency upgrades
- CI-related changes within a repo

**When to use:**
- Tier 1 tasks (auto-execute) — docs, version bumps, test additions
- Tier 2 tasks (plan-then-execute) — single-repo features, providers, middleware
- Any task that Claude Code has scoped into a clear, self-contained issue
- Batch execution of multiple independent tasks in parallel

**Input contract:** Codex receives either:
- A GitHub Issue with structured body (generated by Claude Code via issue templates)
- A handoff prompt with explicit context: repo, branch, what to do, acceptance criteria, governing ADRs

**Does not:** Make architectural decisions. Modify contracts without explicit instruction. Work across repo boundaries in a single task.

---

### 3. GitHub Copilot — Code in the IDE

**Where:** VS Code / JetBrains, inside a specific repo
**Role:** The surgeon. For when you need to open the hood and work at the code level — debugging, nuanced implementation, exploratory coding, complex refactors.

**When to use:**
- Debugging a specific issue that requires reading and stepping through code
- Implementation that requires iterative design decisions at the code level
- Complex refactors where you need to see the full file context
- Reviewing Codex PRs and making adjustments
- Exploratory work — "let me try this approach and see"
- Anything where the developer needs to be in the loop line-by-line

**Does not:** Plan cross-repo work. Generate work items. Reason about Grid topology.

## Flow by Tier

### Tier 1 — Auto-Execute

```
Developer (intent)
  → Claude Code (scope + work item)
    → Codex (execute, open PR)
      → PR review (automated checks + developer glance)
        → Merge
```

Examples: doc fixes, test additions, version bumps, changelog updates.
Developer involvement: minimal. Claude Code scopes it, Codex ships it.

### Tier 2 — Plan-Then-Execute

```
Developer (intent)
  → Claude Code (decompose into tasks, generate issues)
    → Developer reviews plan
      → Codex (execute each task, open PRs)
        → Developer reviews PRs
          → Merge
```

Examples: new feature in one repo, new provider, CI workflow change.
Developer involvement: reviews the plan and the PRs. May jump into Copilot for specific PRs.

### Tier 3 — Architecture Review

```
Developer (design question)
  → Claude Code (architecture discussion, ADR draft)
    → Developer reviews and accepts ADR
      → Claude Code (decompose into Tier 2 tasks)
        → Codex (execute tasks in dependency order)
          → Developer reviews PRs
            → Merge upstream first, then downstream
```

Examples: new contract in Abstractions, new Node, context model changes.
Developer involvement: drives the architecture discussion, reviews ADR, reviews execution order, reviews PRs. May use Copilot for upstream contract changes.

## Handoff Protocol: Claude Code → Codex

When Claude Code generates work for Codex, the handoff must include:

```markdown
## Task
{imperative description of what to do}

## Target
- Repo: {repo name}
- Branch: create from `main`

## Context
- Goal: {parent goal title}
- Feature: {parent feature title}
- ADRs: {list of governing ADR IDs, if any}

## Acceptance Criteria
{observable conditions for completion — tests pass, contract implemented, etc.}

## Dependencies
{list of PRs or tasks that must merge before this one starts, if any}

## Constraints
{repo boundaries, invariants, or conventions that apply}
```

This can be delivered as:
- A GitHub Issue body (generated via work item template)
- A direct prompt to Codex with the above structure

## ADR-0044 D3/D6 Codex Authoring Discipline

When Codex executes a packet, ADR-0044 D3 requires the review rubric to be applied upstream before producing the diff. The handoff should keep this checklist visible so Codex authors against the same standard the review agent later evaluates.

Apply the brief ADR-0044 D3 authoring checklist before producing a diff:

- **1. Correctness and functional integrity** — the change satisfies the packet/intent and handles edge cases.
- **3. Maintainability** — the implementation is readable and locally understandable.
- **5. SOLID and design principles** — responsibilities stay cohesive; no speculative abstraction.
- **4. Reuse and ecosystem cohesion** — reuse existing Grid patterns; avoid duplicate helpers/policy.
- **9. Security** — no secret leakage, auth bypass, or widened blast radius.
- **6. Performance and scalability** — no avoidable hot-path, cost, or unbounded-work risk.
- **11. Testing quality** — verification is meaningful and covers the changed behavior.
- **18. AI and agent-specific concerns** — authorship, idempotency, context, and replay risks are explicit.
- **20. Human factors** — PRs are easy for Oleg to review and do not hide operational trade-offs.

The remaining D3 categories are still in scope by reference through `.claude/agents/review.md`; do not treat this short list as exhaustive.

Per ADR-0044 D6, Codex-authored PRs must declare authorship in both places:

- PR body line: `Authorship: agent-codex`
- Commit trailer: `Authorship: agent-codex`

When Codex materially shares authorship with another surface, use `Authorship: mixed`. Use only the five D6 classes: `human`, `agent-codex`, `agent-copilot`, `agent-claude-code`, or `mixed`.

## Handoff Protocol: Codex → Developer (PR Review)

When Codex opens a PR, the developer evaluates:

1. **Does the PR match the issue?** — Implementation aligns with acceptance criteria
2. **Do checks pass?** — CI green, canary tests pass
3. **Does it respect boundaries?** — No code that belongs in another Node
4. **Does it preserve invariants?** — No violations from `invariants.md`

If adjustments are needed:
- Minor fixes → developer uses Copilot in IDE to adjust, pushes to the PR branch
- Fundamental rework → close PR, refine the issue, re-assign to Codex or handle manually

## When to Break the Flow

The three-surface flow is the default. Break it when:

| Situation | Action |
|-----------|--------|
| Codex produces a PR that needs subtle fixes | Open IDE, use Copilot to adjust, push to same branch |
| Task is too nuanced for Codex to handle autonomously | Skip Codex, implement directly with Copilot |
| Debugging a production issue | Start in IDE with Copilot, escalate to Claude Code if it's systemic |
| Exploratory prototype | Start in IDE with Copilot, bring to Claude Code when ready to formalize |
| Emergency hotfix | IDE + Copilot, skip the planning layer, backfill the issue later |

The surfaces are tools, not a rigid pipeline. The developer routes work to the surface that fits.
